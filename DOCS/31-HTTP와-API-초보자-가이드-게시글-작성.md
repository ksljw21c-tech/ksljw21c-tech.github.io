### 본문 구조

1. **H1: 제목 및 전체 개요**
   - HTTP와 API의 중요성
   - 웹의 모든 것이 HTTP와 API를 통해 이루어짐
   - 전화 통화 비유 (전화선=HTTP, 전화번호=API)
   - 웹 개발의 핵심 개념
   - 글의 목적과 구성 설명

2. **H2-1: HTTP란 무엇인가 - 웹 통신의 기본 규칙**
   - HTTP의 정의
     - HyperText Transfer Protocol의 약자
     - 웹에서 정보를 주고받기 위한 규칙
     - 클라이언트와 서버가 소통하기 위한 약속
   - HTTP의 역사
     - 1990년대 초 개발
     - HTTP/1.1, HTTP/2, HTTP/3 발전
     - 기본 원리는 변하지 않음
   - HTTP의 작동 방식
     - 요청(Request)과 응답(Response) 패턴
     - 식당 주문 비유
   - HTTP의 특성
     - 무상태성(Stateless)
     - 텍스트 기반 프로토콜
   - HTTPS
     - HTTP에 암호화 추가
     - 보안의 기본 요구사항
     - 2025년 현재 대부분의 웹사이트가 사용
   - HTTP 요청 구조
     - HTTP 메서드
     - URL
     - 헤더(Headers)
     - 본문(Body)
   - HTTP 응답 구조
     - 상태 코드(Status Code)
     - 응답 헤더
     - 응답 본문
     - 다양한 데이터 형식 (HTML, JSON, XML)

3. **H2-2: API란 무엇인가 - 애플리케이션 간의 대화 창구**
   - API의 정의
     - Application Programming Interface의 약자
     - 서로 다른 애플리케이션이 소통할 수 있게 해주는 인터페이스
     - 한 프로그램이 다른 프로그램의 기능을 사용할 수 있게 해주는 창구
   - API의 비유
     - 식당의 주문 창구
     - 외부 요청을 내부 시스템에 전달
   - API의 범위
     - 웹 개발뿐만 아니라 모든 소프트웨어 개발
     - 운영체제, 라이브러리, 클라우드 서비스
     - 웹 API (REST API, GraphQL API, SOAP API)
   - API의 장점
     - 재사용성
     - 여러 애플리케이션에서 사용 가능
     - 다양한 플랫폼 지원
   - API의 실전 활용
     - 날씨 정보 API 예시
     - 개발 속도 향상
     - 복잡한 기능의 쉬운 구현
   - 2025년 API 경제
     - 수많은 기업이 서비스를 API로 제공
     - 개발자들이 API를 조합하여 새로운 서비스 생성
   - API의 핵심 개념
     - 엔드포인트(Endpoint)
     - 리소스 접근
     - HTTP 메서드 사용
   - API 보안
     - 인증(Authentication)
     - 권한 부여(Authorization)
     - API 키, OAuth, JWT
   - API 버전 관리
     - URL에 버전 포함
     - 하위 호환성 보장
   - API 문서화
     - Swagger, OpenAPI
     - 테스트 인터페이스 제공
   - API 설계 원칙
     - 일관성
     - 명확성
     - 예측 가능성
   - API 성능 최적화
     - 캐싱
     - 페이징
     - 속도 제한

4. **H2-3: HTTP 메서드 완벽 이해하기 - GET, POST, PUT, DELETE**
   - HTTP 메서드의 역할
     - 클라이언트가 서버에게 작업을 알려주는 명령어
     - CRUD 작업과 직접 연결
   - CRUD와 HTTP 메서드 매핑
     - GET = Read (읽기)
     - POST = Create (생성)
     - PUT = Update (수정)
     - DELETE = Delete (삭제)
   - GET 메서드
     - 서버에서 데이터를 가져오는 데 사용
     - 안전하고 멱등성(Idempotent)을 가짐
     - 쿼리 파라미터 사용
     - 캐싱 가능
     - 본문을 가지지 않음
   - POST 메서드
     - 새로운 리소스를 생성하는 데 사용
     - 멱등성을 가지지 않음
     - 본문에 데이터 포함 (일반적으로 JSON)
     - 특정 작업 수행에도 사용
     - 201 Created 상태 코드
     - 캐싱이 어려움
   - PUT 메서드
     - 기존 리소스를 완전히 교체
     - 멱등성을 가짐
     - 리소스의 전체를 교체
     - Upsert 기능
     - 200 OK 또는 204 No Content
   - PATCH 메서드
     - PUT과 비슷하지만 리소스의 일부만 업데이트
     - PUT보다 더 효율적
   - DELETE 메서드
     - 리소스를 삭제하는 데 사용
     - 멱등성을 가짐
     - 일반적으로 본문을 가지지 않음
     - 소프트 삭제(Soft Delete) 구현 가능

5. **H2-4: REST API의 핵심 원칙과 실전 활용**
   - REST의 정의
     - Representational State Transfer의 약자
     - 웹 서비스를 설계하기 위한 아키텍처 스타일
   - REST의 핵심 원칙
     - 리소스 중심 설계
     - 무상태성(Statelessness)
     - 균일한 인터페이스
     - 캐시 가능성(Cacheability)
     - 클라이언트-서버 분리
   - 리소스 중심 설계
     - 모든 것이 리소스
     - 각 리소스는 고유한 URI로 식별
     - 리소스는 명사로 표현, 동사는 HTTP 메서드로 표현
     - 계층 구조 표현
   - 무상태성(Statelessness)
     - 각 요청이 독립적
     - 서버가 클라이언트의 상태를 저장하지 않음
     - 확장성 향상
     - 토큰이나 쿠키를 통한 상태 관리
   - 균일한 인터페이스
     - 모든 리소스가 동일한 방식으로 접근
     - 표준화된 HTTP 메서드, 상태 코드, 헤더
     - 일관된 사용
   - 캐시 가능성
     - GET 요청은 캐시 가능
     - 적절한 캐시 헤더 포함
     - POST, PUT, DELETE 후 캐시 무효화
   - 클라이언트-서버 분리
     - 클라이언트: 사용자 인터페이스에 집중
     - 서버: 비즈니스 로직과 데이터 저장에 집중
     - 독립적 개발과 배포
     - 다양한 클라이언트 지원
   - REST API 모범 사례
     - 일관된 네이밍 규칙
     - 적절한 HTTP 상태 코드 사용
     - 명확한 에러 처리
     - API 버전 관리
     - 페이징, 필터링, 정렬 기능 제공
   - 2025년 REST API 현황
     - 여전히 가장 널리 사용되는 API 설계 방식
     - GraphQL, gRPC 같은 대안 존재
     - REST의 단순성과 표준화된 접근 방식 선호
