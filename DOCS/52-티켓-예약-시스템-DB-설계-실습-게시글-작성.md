# 52. 티켓 예약 시스템 DB 설계 실습 게시글 작성

## 작업 개요
- **주제**: 티켓 예약 시스템 DB 설계(좌석 락·홀드, 결제/환불/정산, 운영 체크리스트)
- **구조**: H1 p, H2 p, H2 p, H2 p (서술형)
- **목표 글자수**: 5,000자 / **실제**: 약 5,020자
- **용도**: 애드센스 승인용 고품질 콘텐츠
- **도구**: tavily-mcp 최신 자료 수집

## 수집된 자료 분석
- Ticketmaster/티켓팅 시스템 설계 글: 좌석 홀드 TTL, 분산 락, 만료 복구 강조
- Redis 기반 좌석 홀드 예시: TTL로 임시 예약, 만료 시 자동 해제 패턴
- GeeksforGeeks 예약 DB 설계 가이드: 엔티티·관계·정규화 기본 흐름 정리

## 작성된 글 핵심
- 엔티티/관계: 사용자, 이벤트, 회차, 좌석(회차별 스냅샷), 예약, 결제, 환불, 정산, 쿠폰/프로모션, 운영 로그
- 좌석 흐름: HOLD→BOOKED, 만료 자동 해제, Redis/DB 락 병행, 가격 스냅샷 보존
- 결제/환불/정산: 결제 상태 이력, 환불 1:N, 회차 매출·주최자 정산 분리
- 운영 체크리스트: 인덱스·파티셔닝·캐싱, FK/UNIQUE로 중복 차단, 모니터링 지표(락 실패율, 만료 회수, 결제 실패율 등), 보안/PII/규제 대응, 부하 테스트와 복구 리허설

## 글자 수
- 약 5,020자 (목표 충족)
